# 系统需求分析
## 系统需求类型
### 系统功能需求
* 功能需求里可以有通用模块，如系统管理模块、警告模块

### 系统非功能需求
| 项 | 内容 | 说明 |
| - | - | - |
| 接口 | 用户接口, 软件接口(API/对外函数)，硬件接口(通讯/IO) | 用户接口是对“UI质量”的约束和保障，是标准，如易用性(操作效率、容错率)、可访问性、一致性 |
| 安全 |  | 故障树分析法（FTA） |
| 性能 |  |  |
| 合规 | 标准法规，网络安全等 | 审计日志，版权保护，自动注销(超时自动退出)，数据备份与灾难恢复 |
| 可用 |  |  |

| 兼容 |  |  |

## 上游输入依据
1. 业务目标、产品需求
1. 技术约束：技术可行性。如系统内现在使用技术的约束(如只支持有线网络)，现有技术的约束
1. 外部环境约束：法规，需对接的系统

## 分析方法
1. 产品需求对应流程的每个步骤环节拆解成“系统可执行、可验证”的最小单元“原子需求”。
1. 流程需覆盖“正常操作场景”和“异常处理场景”。
1. 每个拆解出的需求都对应“用户的一个具体操作”或“系统的一个处理”。遵循SMART原则。如"支持用户2秒内查询近6个月的历史订单"。
